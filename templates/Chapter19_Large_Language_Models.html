Chapter 19: Large Language Models (LLMs)

1. Introduction to Large Language Models

Large Language Models (LLMs) are advanced AI systems trained on vast amounts of text data to understand, generate, summarise, translate, and reason using human language. They represent the current state of the art in Natural Language Processing and form the backbone of modern Generative AI.

LLMs form the foundation of systems such as ChatGPT, Gemini, Claude, and Llama. What makes them "large" is the combination of massive training datasets (trillions of tokens), enormous model sizes (billions to trillions of parameters), and unprecedented compute used during training.

Core mechanism: LLMs predict the next most likely token (word or word-fragment) based on all preceding context. This simple objective — repeated billions of times — gives rise to emergent capabilities like reasoning, translation, code generation, and creative writing.

What Makes LLMs Different from Earlier NLP

Scale

GPT-4 has an estimated 1 trillion+ parameters. Earlier models had millions. This scale enables emergent abilities not seen in smaller models.

Few-Shot Learning

LLMs can learn a new task from just 2–3 examples in the prompt — no retraining required. Earlier models needed thousands of labelled examples.

Generality

A single LLM handles translation, coding, summarisation, maths, and conversation — tasks that previously required separate specialised models.

Emergent Reasoning

Beyond memorisation, large enough models exhibit genuine multi-step reasoning, analogy, and problem-solving abilities that were not explicitly trained.

2. Evolution of Language Models

Language modelling has gone through five distinct generations over six decades, each driven by a breakthrough in either architecture, data, or compute.

EraGenerationKey IdeaLimitation
1950s–1980sRule-Based SystemsHandcrafted grammar rules and dictionaries (e.g. ELIZA)Brittle — couldn't handle anything outside the rules
1990s–2000sStatistical Models (N-grams)Predict next word using frequency counts from a corpusNo semantic understanding; exponential data growth needed for longer context
2013–2017Neural Networks (Word2Vec, RNN, LSTM)Word embeddings capture semantics; RNNs process sequencesSequential processing limits parallelism; vanishing gradient for long sequences
2017–2018Transformers (BERT, GPT-1)Self-attention replaces recurrence; parallel processing; pre-training + fine-tuningModels still relatively small; fine-tuning required per task
2019–presentLarge Language Models (GPT-3/4, Claude, Gemini)Scale laws: bigger model + more data = better performance; few-shot and zero-shot capabilityHigh cost, hallucinations, alignment challenges
The Scaling Hypothesis: Researchers at OpenAI and DeepMind discovered that LLM performance improves predictably as you scale model size, training data, and compute — a relationship known as the "Chinchilla scaling laws."

3. Tokens and Tokenization

Before any text is processed by an LLM, it must be converted into tokens — the basic units the model operates on. A token is not always a complete word; it can be a word, a syllable, a punctuation mark, or a common sub-word fragment.

Sentence: "Artificial Intelligence is amazing."

Tokens (approximate):
["Art", "ificial", " Intelligence", " is", " amazing", "."]

Each token is mapped to a unique integer ID in the model's vocabulary.
"Art"         → 8502
"ificial"     → 9542
" Intelligence" → 22478
...

Why Sub-word Tokenization?

Handles Unknown Words

"Tokenization" breaks into "Token" + "ization" — both known sub-words. The model can process new compound words without an "unknown" token.

Vocabulary Efficiency

A vocabulary of ~50,000 sub-word tokens covers virtually all English text. A word-level vocabulary would need millions of entries.

Multilingual

Sub-word tokenizers like BPE (Byte-Pair Encoding) and SentencePiece work across languages without separate per-language vocabularies.

Tokenization Algorithms

AlgorithmUsed ByHow It Works
Byte-Pair Encoding (BPE)GPT-2, GPT-4, LlamaIteratively merges the most frequent adjacent byte pairs into new tokens
WordPieceBERT, GeminiSimilar to BPE but chooses merges that maximise training data likelihood
SentencePieceT5, Llama 2, MistralLanguage-agnostic; works directly on raw text without pre-tokenisation
TiktokenGPT-3.5, GPT-4OpenAI's fast BPE implementation optimised for speed and consistency
Token cost matters: Most LLM APIs charge per token. 1 token ≈ 0.75 English words on average. A 1,000-word document is roughly 1,300–1,500 tokens. Understanding tokenisation helps estimate API costs and context window usage.

4. Embeddings

Embeddings convert tokens, sentences, or entire documents into dense numerical vectors (arrays of floating-point numbers) that capture semantic meaning. Words with similar meanings are mapped to nearby points in this high-dimensional vector space.

Word vector examples (simplified to 3 dimensions):

"King"   → [0.92, 0.10, 0.74]
"Queen"  → [0.89, 0.11, 0.71]
"Man"    → [0.91, 0.08, 0.42]
"Woman"  → [0.88, 0.09, 0.40]

Famous relationship:
Vector("King") − Vector("Man") + Vector("Woman") ≈ Vector("Queen")

Types of Embeddings

TypeScopeExample ModelsUse Case
Word EmbeddingsSingle wordWord2Vec, GloVe, FastTextClassic NLP tasks, similarity search
Sentence EmbeddingsSentence or paragraphSentence-BERT, E5, text-embedding-3Semantic search, document retrieval
Contextual EmbeddingsToken in contextBERT, GPT hidden statesNamed entity recognition, classification
Multimodal EmbeddingsText + image/audioCLIP, ImageBindCross-modal search, image captioning

Applications of Embeddings

Semantic Search

Instead of exact keyword matching, embeddings find results that mean the same thing even if they use different words.

Recommendation Systems

Embed user preferences and item descriptions into the same space. Recommend items whose vectors are close to the user's preference vector.

Vector Databases

Tools like Pinecone, Weaviate, and pgvector store and index millions of embedding vectors for fast nearest-neighbour retrieval.

RAG (Retrieval-Augmented Generation)

Document chunks are embedded and stored. At query time, the most semantically similar chunks are retrieved and injected into the LLM prompt.

Clustering & Classification

Embeddings allow unsupervised clustering of documents by topic, or few-shot classification using cosine similarity instead of a trained classifier.

Anomaly Detection

Unusual text (fraud, abuse, outliers) appears far from normal clusters in embedding space and can be flagged automatically.

5. Transformer Architecture

The Transformer, introduced in the 2017 paper "Attention Is All You Need" (Vaswani et al.), replaced recurrent networks with self-attention, enabling full parallelism during training and direct modelling of relationships between any two tokens regardless of their distance.

Input Tokens
      ↓
Token Embeddings + Positional Encoding
      ↓
┌────────────────────────────────────┐
│  Transformer Block × N             │
│  ┌──────────────────────────────┐  │
│  │ Multi-Head Self-Attention    │  │
│  │ + Residual Connection + LN  │  │
│  └──────────────────────────────┘  │
│  ┌──────────────────────────────┐  │
│  │ Feed-Forward Network (FFN)   │  │
│  │ + Residual Connection + LN  │  │
│  └──────────────────────────────┘  │
└────────────────────────────────────┘
      ↓
Output Logits (next-token probabilities)

Key Components

ComponentPurpose
Token EmbeddingConverts each token ID into a dense vector of size d_model (e.g. 4096 for LLaMA-7B)
Positional EncodingInjects token position information — attention itself is order-agnostic
Multi-Head Self-AttentionEach token attends to all other tokens across multiple parallel "heads"
Feed-Forward Network (FFN)Two linear layers with a non-linearity; applies token-wise transformations
Residual ConnectionsSkip connections (x + sublayer(x)) prevent vanishing gradients in deep networks
Layer NormalisationStabilises activations within each block; applied before (Pre-LN) in modern models

Encoder vs Decoder vs Encoder-Decoder

Encoder-Only (BERT)

Reads the full input bidirectionally. Best for understanding tasks: classification, NER, semantic search. Not used for text generation.

Decoder-Only (GPT, LLaMA, Claude)

Generates text autoregressively — one token at a time, left to right. The dominant architecture for modern LLMs and chat models.

Encoder-Decoder (T5, BART)

Encoder processes the input; decoder generates the output. Used for translation, summarisation, and question answering.

6. Attention Mechanism

Attention is the core innovation of the Transformer. It allows every token to directly "look at" every other token in the sequence and decide how much weight to assign each one when building its own representation.

Attention(Q, K, V) = softmax( Q·Kᵀ / √dₖ ) · V

Where:
  Q = Query matrix  (what am I looking for?)
  K = Key matrix    (what do I contain?)
  V = Value matrix  (what do I pass forward?)
  dₖ = dimension of key vectors (scaling factor)

Intuition Example

"The cat sat on the mat because it was tired."

When generating a response, the model must resolve "it".
The attention mechanism assigns high weight to "cat" 
when processing "it" — learning this coreference
from billions of training examples.

Multi-Head Attention

Instead of one attention operation, the model runs attention in parallel across multiple "heads" (e.g. 32 heads for LLaMA-7B). Each head can learn to attend to different types of relationships — syntax, coreference, topic, position.

Types of Attention Used in LLMs

TypeDescriptionUsed By
Full Self-AttentionEvery token attends to every other. O(n²) memory cost.Original GPT, BERT
Causal (Masked) AttentionEach token only attends to previous tokens. Required for autoregressive generation.GPT-4, LLaMA, Claude
Grouped Query Attention (GQA)Multiple query heads share a single key-value head. Reduces memory at inference.LLaMA-2/3, Gemma
Flash AttentionHardware-optimised attention that avoids materialising the full attention matrix in VRAM.Most modern LLMs
Why attention beats RNNs: RNNs process tokens one-by-one sequentially — information from 50 steps ago gets diluted through many multiplications. Attention gives every token direct access to every other token in O(1) steps, regardless of distance.

7. Context Windows

The context window is the maximum number of tokens an LLM can process in a single forward pass — both the input prompt and the generated output combined. Everything outside the window is invisible to the model.

ModelContext WindowApprox. Pages of Text
GPT-3 (2020)4,096 tokens~6 pages
GPT-4 (2023)128,000 tokens~200 pages
Claude 3 (2024)200,000 tokens~500 pages
Gemini 1.5 Pro (2024)1,000,000 tokens~1,500 pages
LLaMA-3 (2024)128,000 tokens~200 pages

Why Context Window Size Matters

Long Document Analysis

A 200K token window lets you feed entire books, legal contracts, or codebases into a single prompt for analysis.

Multi-Turn Conversations

Longer context means the model remembers earlier parts of a conversation without the chat history being truncated.

RAG Reduction

With very large windows, you can inject entire knowledge bases directly, reducing the need for retrieval pipelines.

Cost and Latency

Attention scales as O(n²) with sequence length. Larger context = more compute and higher API cost per call.

"Lost in the Middle" problem: Research shows LLMs perform best on information at the start and end of the context window. Information buried in the middle of very long contexts is less reliably recalled — even when technically within the window.

8. Training Large Language Models

Training an LLM is a multi-stage, massively distributed process that requires petabytes of data, thousands of GPUs, and months of computation. The objective is simple — predict the next token — but the scale makes it one of the most expensive engineering endeavours in history.

Estimated ~1T parameters (mixture of experts). Training cost estimated at $50–100M USD.

Key bottleneck

GPU memory (VRAM) limits batch size. Distributed training across hundreds of nodes requires high-speed interconnects (NVLink, InfiniBand).

9. Fine-Tuning

Fine-tuning takes a pre-trained LLM and continues training it on a smaller, task-specific or domain-specific dataset. This adapts the model's behaviour without training from scratch, saving enormous compute cost.

Fine-Tuning Methods

MethodDescriptionCostBest For
Full Fine-TuningAll model weights are updated on the new datasetVery HighLarge organisations with dedicated GPU clusters
LoRA (Low-Rank Adaptation)Trains small low-rank matrices added to existing weight matrices. Only ~0.1% of parameters trained.LowMost practical fine-tuning tasks on consumer GPUs
QLoRALoRA applied to a 4-bit quantised model. Further reduces VRAM requirements.Very LowFine-tuning 7B–13B models on a single 24GB GPU
Instruction Fine-TuningTrains on (instruction, response) pairs to make the model follow natural language instructionsMediumBuilding chat assistants and instruction-following models
PEFT (Parameter-Efficient FT)Umbrella term for methods that train only a small fraction of parameters (LoRA, prefix tuning, adapters)Low–MediumDomain adaptation with limited resources

Domain Fine-Tuning Examples

Medical Chatbot

Fine-tuned on clinical notes, medical literature, and Q&A pairs to provide accurate symptom guidance and drug information.

Legal Assistant

Fine-tuned on case law, contracts, and statutes to draft legal documents and answer jurisdiction-specific questions.

Customer Support

Fine-tuned on company FAQs and past support tickets to handle product-specific queries consistently.

Financial Analysis

Fine-tuned on earnings reports, SEC filings, and analyst notes to interpret financial language and generate summaries.

Code Assistant

Fine-tuned on proprietary codebases to suggest completions and explain internal APIs not seen during pre-training.

Language Localisation

Fine-tuned on low-resource language data (e.g. Tamil, Swahili) to improve performance in languages underrepresented in pre-training.

10. RLHF and Alignment

A raw pre-trained LLM predicts the statistically likely next token — but "likely" is not the same as "helpful, harmless, and honest." Alignment techniques shape model behaviour to match human values and follow instructions reliably.

Reinforcement Learning from Human Feedback (RLHF)

1

Supervised Fine-Tuning (SFT)

Human trainers write example responses to prompts. The model is fine-tuned on these high-quality demonstrations to produce instruction-following behaviour.

2

Reward Model Training

Human raters rank multiple model responses to the same prompt. A separate "reward model" is trained to predict human preference scores.

3

PPO Optimisation

The LLM is optimised using Proximal Policy Optimisation (PPO) — a reinforcement learning algorithm — to generate responses that maximise the reward model's score.

DPO (Direct Preference Optimisation): A newer, simpler alternative to RLHF that skips the reward model entirely. The LLM is directly fine-tuned on (preferred, rejected) response pairs. Used by Llama-3 and Mistral fine-tunes. Computationally cheaper and often equally effective.

Alignment Objectives

PropertyMeaningExample Failure Without It
HelpfulActually answers the user's question fully and accuratelyGiving vague non-answers to avoid any risk
HarmlessRefuses to produce dangerous, abusive, or illegal contentGenerating instructions for harmful activities
HonestAcknowledges uncertainty; doesn't fabricate or deceiveConfidently stating false facts (hallucination)

11. Inference

Inference is the process of using a trained model to generate outputs for new inputs. Every time you send a message to ChatGPT or Claude, inference is happening. Unlike training (which updates weights), inference is a read-only forward pass.

User Prompt: "Explain Machine Learning in simple terms."
      ↓
Tokenize prompt
      ↓
Forward pass through all transformer layers
      ↓
Softmax over vocabulary → probability distribution
      ↓
Sample next token (e.g. "Machine")
      ↓
Append token → repeat until [END] token or max length
      ↓
Detokenize → return text to user

Inference Parameters

ParameterEffectTypical Range
TemperatureControls randomness. 0 = deterministic (most likely token); 1 = creative; >1 = chaotic.0.0 – 2.0
Top-p (nucleus sampling)Samples from the smallest set of tokens whose cumulative probability ≥ p. Balances diversity and quality.0.8 – 1.0
Top-kRestricts sampling to the k most likely next tokens.10 – 100
Max tokensMaximum number of tokens to generate in a single response.128 – 4096
Frequency penaltyReduces the probability of tokens that have already appeared, discouraging repetition.0.0 – 2.0

Inference Optimisation Techniques

Quantisation

Reduce model weights from 32-bit or 16-bit floats to 8-bit or 4-bit integers. Cuts VRAM usage by 4×–8× with minimal quality loss.

KV Cache

Caches the key-value attention matrices for the prompt, avoiding recomputation on each new token generation step.

Speculative Decoding

A small draft model proposes multiple tokens; the large model verifies them in parallel. Speeds up inference 2–3×.

Batching

Processing multiple requests simultaneously on the same GPU to improve hardware utilisation and throughput.

13. Open Source vs Closed Source Models

Choosing between open-weight and closed (API-only) models involves trade-offs across cost, control, privacy, and ease of use.

✅ Open-Weight Models (LLaMA, Mistral, Gemma)

  • Weights are publicly downloadable
  • Can be self-hosted on your own infrastructure
  • Full control over data — nothing leaves your servers
  • Can be fine-tuned, quantised, and modified freely
  • No per-token API cost (pay only for compute)
  • Requires ML engineering expertise to deploy and maintain
  • Security and updates are your responsibility

🔒 Closed / API Models (GPT-4, Claude, Gemini)

  • Access via REST API — no infrastructure needed
  • Managed updates, scaling, and uptime by provider
  • State-of-the-art performance, especially on complex reasoning
  • Pay-per-token pricing — cost scales with usage
  • Data sent to provider's servers (privacy considerations)
  • No access to model weights; cannot fine-tune base model
  • Dependent on provider's pricing, availability, and policies

Decision Guide

ScenarioRecommended Approach
Prototype / proof of conceptClosed API (fastest to get started)
Sensitive data / on-premises requirementOpen-weight, self-hosted
Domain-specific fine-tuning neededOpen-weight (LoRA / QLoRA)
Maximum performance, budget availableClosed API (GPT-4o / Claude 3.5)
High-volume, cost-sensitive productionOpen-weight or smaller closed models
Edge / mobile deploymentOpen-weight quantised (Phi-3, Gemma 2)

14. Retrieval-Augmented Generation (RAG)

RAG is an architecture that enhances LLM responses by retrieving relevant documents from an external knowledge base at query time and injecting them into the prompt. This grounds the model in verified, up-to-date information and dramatically reduces hallucinations.

User Query
      ↓
Embed query → query vector
      ↓
Search vector database (cosine similarity)
      ↓
Retrieve top-k relevant document chunks
      ↓
Inject chunks into LLM prompt as context
      ↓
LLM generates answer grounded in retrieved documents
      ↓
Response to user (with source citations)

RAG Components

Document Chunking

Source documents are split into overlapping chunks (e.g. 512 tokens with 50-token overlap) to preserve context across chunk boundaries.

Embedding Model

Each chunk is converted to a vector using an embedding model (e.g. text-embedding-3-small, E5-large). Stored in the vector DB.

Vector Database

Stores and indexes embedding vectors for fast approximate nearest-neighbour search. Examples: Pinecone, Weaviate, Chroma, pgvector.

Retriever

At query time, embeds the user's question and retrieves the top-k most similar document chunks from the vector DB.

Prompt Assembly

Retrieved chunks are formatted and injected into the LLM prompt as context, along with the user's question and instructions.

Generator (LLM)

Answers the question using only the provided context. Often instructed: "If the answer is not in the context, say so."

RAG vs Fine-Tuning: Use RAG when knowledge changes frequently (news, product docs, regulations). Use fine-tuning when you want the model to change its style, tone, or domain vocabulary. For best results, combine both.

15. Applications of LLMs

LLMs are being deployed across virtually every industry. Below are the most impactful use cases in production today.

Conversational AI

Customer service bots, HR assistants, and virtual agents that handle complex multi-turn conversations with context retention.

Code Generation & Review

GitHub Copilot, Cursor, and Claude in IDEs autocomplete code, explain functions, generate tests, and review PRs.

Content Generation

Marketing copy, blog posts, product descriptions, and social media content generated at scale from structured briefs.

Document Summarisation

Condensing legal contracts, research papers, earnings calls, and meeting transcripts into concise summaries.

Language Translation

High-quality translation across 100+ languages, including nuanced domain-specific terminology (medical, legal, technical).

Question Answering

Enterprise search systems where employees ask questions about internal documents, policies, and knowledge bases.

Medical & Clinical AI

Summarising patient records, suggesting differential diagnoses, and drafting clinical notes from doctor-patient conversations.

Education & Tutoring

Personalised tutoring that adapts explanations to the student's level, generates practice problems, and gives instant feedback.

Data Extraction

Extracting structured data (names, dates, entities, relationships) from unstructured documents like invoices, forms, and reports.

Software Testing

Generating unit tests, edge case scenarios, and regression test suites from code or specification documents.

16. Limitations of LLMs

Despite their impressive capabilities, LLMs have significant and well-documented limitations that must be understood before deploying them in production.

LimitationDescriptionMitigation
HallucinationsModel generates confident but factually incorrect informationRAG, grounding prompts, human review, temperature=0
Knowledge CutoffModel has no knowledge of events after training data cutoffRAG with live data sources, web search tools
Bias in ResponsesReflects biases present in training data — demographic, cultural, politicalDiverse training data, RLHF, bias audits
High Computational CostTraining and inference require expensive GPUs and significant energyQuantisation, smaller distilled models, efficient serving
Privacy ConcernsUser prompts may be logged and used for training; PII risksOn-premise deployment, data processing agreements
Lack of ReasoningStruggles with multi-step logical and mathematical reasoning without CoT promptingChain-of-thought prompting, tool use (code execution)
Context Window LimitsCannot process arbitrarily long documents in a single callRAG, chunking strategies, models with larger windows
No Real-World GroundingNo ability to verify claims, check facts, or access real-time informationTool use, web search integration, RAG
InconsistencySame prompt can produce different answers at different temperatures or runsTemperature=0, structured output formats, validation layers
Responsible AI Deployment: Organisations deploying LLMs should implement human-in-the-loop review for high-stakes decisions (medical, legal, financial), establish output validation pipelines, maintain audit logs, and clearly disclose to users when they are interacting with an AI system.

17. Mini Project Idea

Build a Document Question-Answering Assistant using an LLM API and a RAG pipeline that answers questions grounded in company or course documents.

Suggested stack: Python · LangChain or LlamaIndex · OpenAI / Groq API · Chroma or FAISS vector DB · Streamlit for the UI · PyPDF2 or pdfplumber for document parsing.

Project Steps

1

Document Ingestion

Load PDF or text documents using PyPDF2 or pdfplumber. Handle multi-page documents and preserve section structure.

2

Chunking & Embedding

Split documents into 512-token chunks with 50-token overlap. Embed each chunk using text-embedding-3-small (OpenAI) or a local model.

3

Vector Store

Store embeddings in a Chroma or FAISS vector database. Build an index for fast cosine similarity search.

4

Retrieval Pipeline

When a user asks a question, embed it and retrieve the top-3 most similar document chunks from the vector store.

5

Prompt Assembly & Generation

Construct a prompt that includes the retrieved chunks as context and instructs the LLM to answer only from the provided information.

6

Streamlit UI

Build a chat-style interface where users can upload PDFs, ask questions, and see both the answer and the source chunks it was derived from.
