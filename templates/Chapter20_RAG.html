Chapter 20: Retrieval-Augmented Generation (RAG)

1. Introduction to RAG

Retrieval-Augmented Generation (RAG) is an AI architecture that combines the reasoning and language fluency of Large Language Models (LLMs) with the factual precision of external information retrieval systems. Rather than relying solely on knowledge baked into model weights during training, a RAG system dynamically fetches the most relevant information from an external knowledge base at query time — and hands it to the LLM as context before generating a response.

Think of it as the difference between a student answering from memory (standard LLM) and a student taking an open-book exam (RAG). The student with the textbook in front of them gives more accurate, specific, and verifiable answers — and so does a RAG-powered system.

The Core Equation:
RAG = Retrieval (Search Engine) + Generation (LLM Reasoning)

RAG was formally introduced by Lewis et al. (Facebook AI Research, 2020) in the paper "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks".

RAG vs Standard LLM — Side by Side

❌ Standard LLM (Closed-Book)

  • Answers only from training memory
  • No knowledge of events after cutoff date
  • Cannot access private or proprietary data
  • Prone to confident hallucinations on specific facts
  • Updating requires expensive retraining

✅ RAG-Augmented LLM (Open-Book)

  • Answers grounded in retrieved, verified documents
  • Can access live, up-to-date information
  • Works with your private PDFs, databases, and files
  • Can cite sources — answers are auditable
  • Knowledge updated instantly by editing the document store
💬 Discussion: Why is RAG preferred over simply fine-tuning the LLM on your data?

Fine-tuning bakes knowledge into model weights — it is expensive, slow, and produces a model that becomes stale the moment data changes. RAG stores knowledge externally, so updating your knowledge base is as simple as uploading a new PDF. For most business use-cases (product FAQs, HR policies, regulations), RAG is the right tool. Fine-tuning is better for changing style or tone, not for updating facts.

2. Why RAG is Needed

Deploying a raw LLM in a real-world enterprise environment exposes four hard limitations that make it unreliable without augmentation:

ProblemWhat It MeansReal-World Impact
Knowledge CutoffLLMs only know facts up to their training dateA legal assistant trained in 2023 doesn't know 2024 regulation amendments
HallucinationLLMs invent plausible-sounding but false answers when uncertainA medical bot might cite a non-existent drug dosage with full confidence
No Private Data AccessBase models have no knowledge of your proprietary documentsAn HR bot cannot answer "How many leave days do I have?" from a public model
Costly RetrainingUpdating LLM weights requires GPU-hours and thousands of dollarsA product manual changes every month — retraining each time is not feasible
💬 Discussion: Hallucinations are the biggest enterprise blocker. Why do LLMs hallucinate?

LLMs are trained to produce fluent, high-probability text — not to verify facts. When asked about something outside their training distribution, the model does not say "I don't know." Instead, it generates the most statistically likely continuation, which often sounds correct but is fabricated. RAG solves this by providing the model with the actual answer in the context — the model just has to read and rephrase it.

3. RAG Architecture

A production RAG system has two distinct, decoupled pipelines. They run at different times and are maintained independently.

Pipeline 1: Ingestion (Offline, Run Once or on Update)

Raw Documents (PDF, Word, HTML, CSV, Database)
      ↓
Document Loader  →  Extract raw text
      ↓
Text Splitter    →  Chunk into overlapping segments
      ↓
Embedding Model  →  Convert each chunk to a vector
      ↓
Vector Database  →  Store vectors + original text + metadata

Pipeline 2: Retrieval & Generation (Online, Per User Query)

User Question
      ↓
Embedding Model  →  Convert question to vector
      ↓
Vector Database  →  Cosine similarity search → Top-K chunks
      ↓
Prompt Assembly  →  [System Instructions] + [Retrieved Chunks] + [User Question]
      ↓
LLM              →  Generate grounded answer
      ↓
Response to User (with optional source citations)
Key Insight: The ingestion pipeline runs once when documents are added or updated. The retrieval + generation pipeline runs on every user query in milliseconds. This separation is what makes RAG scalable — you can have millions of documents indexed and add new ones without touching the live serving system.

4. Knowledge Bases — Document Sources

A RAG system is only as good as its knowledge base. The documents you ingest determine what the system can and cannot answer. RAG is format-agnostic — it can ingest virtually any text-bearing source.

PDF Documents

Most common source. Product manuals, policy documents, research papers, contracts. Use PyPDF2 or pdfplumber for extraction.

Word / DOCX

HR policies, internal reports, templates. Use python-docx or LangChain's Docx2txtLoader.

Web Pages / HTML

Documentation sites, wikis, blogs. Use LangChain's WebBaseLoader or BeautifulSoup for scraping.

CSV / Excel

Product catalogues, pricing tables, employee directories. Treat each row as a chunk or embed row summaries.

SQL Databases

Query database tables, convert rows to text, embed the results. Useful for dynamic data that changes daily.

Confluence / Notion

Internal knowledge bases. Use official API connectors or LangChain integrations to sync automatically.

Emails / Slack

Historical communications. Embed conversation threads for customer support or project memory use cases.

Code Repositories

GitHub repos for code-aware assistants. Split by function/class rather than character count for best results.

5. Chunking Strategies

Chunking is the process of splitting large documents into smaller, overlapping segments before embedding them. It is arguably the most impactful engineering decision in a RAG system — bad chunking breaks semantic meaning and makes retrieval unreliable.

StrategyHow It WorksBest ForWeakness
Fixed CharacterSplits every N characters regardless of contentQuick prototypingOften cuts sentences mid-way, ruining meaning
Recursive CharacterTries to split at paragraph → sentence → word boundaries in priority orderGeneral-purpose documents (default choice)Chunk sizes can still vary widely
Sentence / ParagraphSplits strictly at punctuation or double newlinesWell-structured prose documentsSome sentences are very long; chunks can be uneven
Semantic ChunkingGroups sentences that are semantically related using embedding similarityComplex documents with mixed topics per pageSlow; requires running embeddings twice
Token-BasedSplits at exactly N tokens (not characters)When you need precise context window controlRequires running the tokenizer first
Document-SpecificSplits by section headers, Markdown headings, or HTML tagsStructured docs like handbooks, wikis, codeRequires document-specific logic

Code Example: Recursive Character Splitter (Recommended)

from langchain.text_splitter import RecursiveCharacterTextSplitter

raw_text = """
Retrieval-Augmented Generation (RAG) is a framework that combines 
information retrieval with language generation. It was introduced by 
Facebook AI Research in 2020.

The system works in two stages: first, relevant documents are retrieved 
from an external knowledge base. Second, the retrieved text is passed 
to a language model to generate a final answer.

RAG significantly reduces hallucinations because the model is explicitly 
given the source text and instructed to answer only from that context.
"""

splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,       # target size in characters
    chunk_overlap=30,     # overlap to preserve cross-chunk context
    separators=["\n\n", "\n", ". ", " ", ""]  # priority split order
)

chunks = splitter.split_text(raw_text)

for i, chunk in enumerate(chunks):
    print(f"\n--- Chunk {i+1} ({len(chunk)} chars) ---")
    print(chunk)

# --- CONSOLE OUTPUT ---
# --- Chunk 1 (197 chars) ---
# Retrieval-Augmented Generation (RAG) is a framework that combines
# information retrieval with language generation. It was introduced by
# Facebook AI Research in 2020.
#
# --- Chunk 2 (214 chars) ---
# The system works in two stages: first, relevant documents are retrieved
# from an external knowledge base. Second, the retrieved text is passed
# to a language model to generate a final answer.
#
# --- Chunk 3 (176 chars) ---
# RAG significantly reduces hallucinations because the model is explicitly
# given the source text and instructed to answer only from that context.

Code Example: Token-Based Splitter

from langchain.text_splitter import TokenTextSplitter

# Split by token count — ensures chunks fit within LLM context windows
token_splitter = TokenTextSplitter(
    chunk_size=150,      # 150 tokens per chunk
    chunk_overlap=20     # 20 token overlap
)

token_chunks = token_splitter.split_text(raw_text)
print(f"Token-based chunks created: {len(token_chunks)}")

# --- CONSOLE OUTPUT ---
# Token-based chunks created: 4
💬 Discussion: What is chunk overlap and why is it needed?

When you split text, important information often spans the boundary between two chunks. For example, a sentence that starts at the end of Chunk 1 and completes at the beginning of Chunk 2 will be split in half — neither chunk captures the full meaning. Overlap (e.g. 50 characters / 20 tokens) copies the tail of each chunk into the beginning of the next, preserving boundary context. A typical production setting is chunk_size=512 tokens with chunk_overlap=50 tokens.

6. Embeddings

Computers cannot understand text — they understand numbers. Embedding models convert each text chunk into a high-dimensional numerical vector (an array of floating-point numbers, typically 768–3072 dimensions). The key property is semantic similarity: chunks that mean similar things are placed close together in this mathematical space, regardless of exact word choice.

# Conceptual vector space (simplified to 2D):
#
#  "dog"    ●──────────────────● "puppy"    (close together — similar meaning)
#
#  "finance"●                              (far from dog/puppy — different topic)
#
# Real embeddings have 1536 dimensions, but the same principle applies.

Code Example: Generating Embeddings with OpenAI

from langchain.embeddings import OpenAIEmbeddings
import numpy as np

embedder = OpenAIEmbeddings(
    model="text-embedding-3-small",  # 1536 dimensions, cheap & fast
    api_key="your_api_key"

# Embed a single query

print(f"Vector dimensions: {len(query_vector)}")
print(f"First 5 values: {query_vector[:5]}")

# Embed multiple documents at once (more efficient)

    "RAG combines retrieval and generation.",
    "Transformers use self-attention mechanisms.",
    "Python is a popular programming language."
]
doc_vectors = embedder.embed_documents(documents)
print(f"\nEmbedded {len(doc_vectors)} documents")

# --- CONSOLE OUTPUT ---
# Vector dimensions: 1536
# First 5 values: [-0.015, 0.022, -0.003, 0.019, 0.041]
# Embedded 3 documents

Code Example: Cosine Similarity — How Retrieval Finds Matches

import numpy as np

def cosine_similarity(vec_a, vec_b):
    """Measures angle between two vectors. 1.0 = identical, 0.0 = unrelated."""
    dot_product = np.dot(vec_a, vec_b)
    norm_a = np.linalg.norm(vec_a)
    norm_b = np.linalg.norm(vec_b)
    return dot_product / (norm_a * norm_b)

# Simulated 4-dimensional vectors for illustration
vec_rag      = np.array([0.9, 0.1, 0.8, 0.2])   # "What is RAG?"
vec_retrieval = np.array([0.88, 0.12, 0.79, 0.21])# "RAG uses retrieval"
vec_python    = np.array([0.1, 0.9, 0.2, 0.8])   # "Python syntax"

sim_rag_retrieval = cosine_similarity(vec_rag, vec_retrieval)
sim_rag_python    = cosine_similarity(vec_rag, vec_python)

print(f"Similarity: 'RAG' vs 'Retrieval doc': {sim_rag_retrieval:.4f}")
print(f"Similarity: 'RAG' vs 'Python doc':    {sim_rag_python:.4f}")

# --- CONSOLE OUTPUT ---
# Similarity: 'RAG' vs 'Retrieval doc': 0.9994   ← HIGH → will be retrieved
# Similarity: 'RAG' vs 'Python doc':    0.0314   ← LOW  → will NOT be retrieved

7. Vector Databases

A standard SQL or NoSQL database stores text and matches it by exact keywords (e.g. WHERE name = 'John'). This cannot find semantically related documents. Vector databases store embedding vectors and use approximate nearest-neighbour (ANN) algorithms to find the most semantically similar vectors in milliseconds, even across millions of entries.

DatabaseTypeBest ForHosting
ChromaDBOpen SourceLocal prototyping, small-scale appsLocal / Docker
FAISSLibrary (Meta)Ultra-fast in-memory search, researchIn-process
PineconeManaged CloudProduction enterprise, no ops burdenCloud (SaaS)
WeaviateOpen Source / CloudHybrid search (keyword + vector)Self-hosted / Cloud

pgvectorPostgreSQL ExtensionTeams already on Postgres, SQL + vectorsAny Postgres host
MilvusOpen SourceBillion-scale vectors, enterpriseSelf-hosted / Cloud

Code Example: ChromaDB — Store and Search Vectors

import chromadb
from chromadb.utils import embedding_functions

# Create a local Chroma client (stores data on disk)
client = chromadb.PersistentClient(path="./chroma_db")

# Use OpenAI embeddings
openai_ef = embedding_functions.OpenAIEmbeddingFunction(
    api_key="your_api_key",
    model_name="text-embedding-3-small"
)

# Create (or load) a collection
collection = client.get_or_create_collection(
    name="company_docs",
    embedding_function=openai_ef
)

# Add documents with metadata
collection.add(
    documents=[
        "Full-time employees receive 15 days of paid annual leave.",
        "Part-time employees receive 7 days of paid annual leave.",
        "Sick leave is capped at 10 days per calendar year for all staff.",
        "Maternity leave entitlement is 26 weeks with full pay.",
    ],
    ids=["doc_001", "doc_002", "doc_003", "doc_004"],
    metadatas=[
        {"source": "HR Handbook", "section": "Leave Policy", "year": 2024},
        {"source": "HR Handbook", "section": "Leave Policy", "year": 2024},
        {"source": "HR Handbook", "section": "Sick Leave",   "year": 2024},
        {"source": "HR Handbook", "section": "Maternity",    "year": 2024},
    ]
)

# Query the collection
results = collection.query(
    query_texts=["How many leave days does a part-time worker get?"],
    n_results=2
)

print("Top Retrieved Chunks:")
for doc, meta, dist in zip(
    results['documents'][0],
    results['metadatas'][0],
    results['distances'][0]
):
    print(f"  Distance: {dist:.4f} | Source: {meta['source']}")
    print(f"  Text: {doc}\n")

# --- CONSOLE OUTPUT ---
# Top Retrieved Chunks:
#   Distance: 0.0821 | Source: HR Handbook
#   Text: Part-time employees receive 7 days of paid annual leave.
#
#   Distance: 0.1543 | Source: HR Handbook
#   Text: Full-time employees receive 15 days of paid annual leave.
💬 Discussion: What is metadata filtering and why does it matter?

Every chunk stored in a vector DB can carry metadata tags — source file, department, date, document type, access level. Before running the expensive vector similarity search, you can pre-filter by metadata: "only search chunks from the 2024 HR handbook." This narrows the search space dramatically (faster + more relevant), and it enables role-based access control — users in the Finance department can only retrieve Finance-tagged chunks.

8. The Retrieval Process

The retrieval step takes the user's raw question and finds the most semantically relevant chunks from the vector database. The quality of what is retrieved determines the quality of the final answer — garbage in, garbage out.

Step-by-Step Retrieval Flow

1

Embed the Query

The user's question is passed through the same embedding model used during ingestion. This produces a query vector of the same dimensionality as the document vectors.

2

Cosine Similarity Search

The query vector is compared against every document vector in the DB using cosine similarity. ANN algorithms (HNSW, IVF) make this sub-millisecond even for millions of vectors.

3

Top-K Retrieval

The K chunks with the highest similarity scores are returned (typically K=3 to K=10). The raw text of these chunks is extracted — not the vectors themselves.

4

Optional Re-ranking

A secondary cross-encoder model re-scores the top-K chunks for even more precise relevance ordering before passing to the LLM. This adds latency but significantly improves answer quality.

Code Example: Retrieval with LangChain Retriever

from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(api_key="your_api_key")

# Load existing Chroma DB from disk
vector_db = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings
)

# Create a retriever — retrieves top 4 most relevant chunks
retriever = vector_db.as_retriever(
    search_type="similarity",       # Options: similarity, mmr, similarity_score_threshold
    search_kwargs={
        "k": 4,                     # number of chunks to return
        "filter": {"year": 2024}    # metadata filter (optional)
    }
)

# Retrieve relevant chunks for a query
query = "What is the annual leave policy for part-time employees?"
relevant_chunks = retriever.get_relevant_documents(query)

for i, chunk in enumerate(relevant_chunks):
    print(f"\nChunk {i+1}:")
    print(f"  Source: {chunk.metadata.get('source', 'Unknown')}")
    print(f"  Content: {chunk.page_content[:150]}...")

# --- CONSOLE OUTPUT ---
# Chunk 1:
#   Source: HR Handbook
#   Content: Part-time employees receive 7 days of paid annual leave per year...
# Chunk 2:
#   Source: HR Handbook
#   Content: Full-time employees receive 15 days of paid annual leave per year...

Retrieval Search Modes Explained

ModeHow It WorksWhen to Use
SimilarityReturns top-K most similar vectors. Pure cosine similarity ranking.Default. Works well for most Q&A tasks.
MMR (Max Marginal Relevance)Balances relevance AND diversity — avoids returning 4 near-identical chunks.When documents are repetitive; good for summarisation tasks.

9. The Generation Process

After retrieval, the relevant chunks are assembled into a carefully engineered prompt and fed to the LLM. The LLM's job is to read the provided context and synthesise a coherent, accurate answer — not to answer from its own memory.

The RAG Prompt Template

SYSTEM_PROMPT = """
You are a helpful enterprise assistant. Your job is to answer the user's 
question using ONLY the information provided in the CONTEXT section below.

Rules:
1. If the answer is clearly stated in the context, answer precisely.
2. If the answer is partially in the context, say what you know and what is unclear.
3. If the answer is NOT in the context, respond: "I don't have that information

4. NEVER invent, guess, or use knowledge outside the context.
5. At the end of your answer, cite the source document(s) used.
"""

Code Example: Full Retrieval + Generation Chain

from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings
from langchain.chat_models import ChatOpenAI
from langchain.prompts import PromptTemplate
from langchain.chains import RetrievalQA

# Setup
embeddings = OpenAIEmbeddings(api_key="your_api_key")
vector_db   = Chroma(persist_directory="./chroma_db", embedding_function=embeddings)
llm         = ChatOpenAI(model_name="gpt-4o", temperature=0)

# Custom RAG prompt that enforces grounded responses
rag_prompt = PromptTemplate(
    input_variables=["context", "question"],
    template="""
You are a precise enterprise assistant.
Answer ONLY from the context below. If the answer is absent, say "I don't know."

CONTEXT:
{context}

QUESTION: {question}

ANSWER:"""
)

# Build the RAG chain
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",                             # Stuff all chunks into one prompt
    retriever=vector_db.as_retriever(search_kwargs={"k": 3}),
    chain_type_kwargs={"prompt": rag_prompt},
    return_source_documents=True                    # Return retrieved sources
)

# Run a query
result = qa_chain({
    "query": "How many paid sick days do part-time employees get?"
})

print("Answer:", result["result"])
print("\nSources used:")
for doc in result["source_documents"]:
    print(f"  - {doc.metadata.get('source')} | Section: {doc.metadata.get('section')}")

# --- CONSOLE OUTPUT ---
# Answer: Part-time employees are entitled to 7 days of paid annual leave per year,
# according to the 2024 HR Handbook. Sick leave is capped at 10 days for all staff.
#
# Sources used:
#   - HR Handbook | Section: Leave Policy
#   - HR Handbook | Section: Sick Leave
💬 Discussion: What does chain_type="stuff" mean and what are the alternatives?

stuff: All retrieved chunks are "stuffed" into one single prompt. Simple and fast. Fails when chunks are too long and exceed the context window.
map_reduce: Each chunk is sent to the LLM separately for a sub-answer, then all sub-answers are combined in a second LLM call. Handles unlimited document length but doubles API cost.
refine: Starts with the first chunk's answer, then iteratively refines it with each subsequent chunk. Slower but produces the most nuanced answers for complex questions.
map_rerank: Generates an answer per chunk and scores each for relevance. Returns the highest-scoring answer. Good for factual precision tasks.

10. Complete RAG Workflow Summary

The full lifecycle from raw documents to a user receiving a cited, accurate answer:

1

Collect Documents

Gather all source files — PDFs, Word docs, web pages, CSVs, database exports. Identify update frequency (daily sync vs once-off).

2

Extract Text

Parse all file formats to raw text. Handle encoding issues, multi-column PDFs, scanned pages (OCR), and tables.

3

Chunk Documents

Split into 256–1024 token segments with 50–100 token overlap. Use Recursive Character Splitter as the default. Attach metadata (source, date, department).

4

Generate Embeddings

Pass every chunk through the embedding model to produce a vector. Cache results to avoid re-embedding unchanged documents on updates.

5

Store in Vector DB

Persist vectors + text + metadata in ChromaDB (local) or Pinecone (production). Build HNSW index for fast retrieval.

6

Receive User Query

Sanitise input. Apply query rewriting if needed (expand abbreviations, fix spelling, decompose multi-part questions).

7

Retrieve Relevant Chunks

Embed query → cosine similarity search → return top-K. Optionally apply metadata filters and re-ranker.

8

Assemble Prompt & Generate

Inject retrieved chunks into prompt template. Call LLM at temperature=0 for factual tasks. Return answer with source citations.

11. Advanced RAG Techniques

Naive RAG (the basic pipeline) works well for simple Q&A. Advanced RAG techniques improve accuracy on complex, multi-hop, or ambiguous queries.

11.1 Query Rewriting

User queries are often vague, misspelled, or use abbreviations. A query rewriter uses an LLM to expand and clarify the query before sending it to the vector DB.

from langchain.chat_models import ChatOpenAI

llm = ChatOpenAI(model_name="gpt-4o", temperature=0)

original_query = "leave days pt staff?"

rewrite_prompt = f"""
Rewrite the following user query to be clear, complete, and suitable 
for searching a corporate HR document. Correct any abbreviations.

Original Query: {original_query}
Rewritten Query:"""

rewritten = llm.predict(rewrite_prompt)
print(f"Original:  {original_query}")
print(f"Rewritten: {rewritten}")

# --- CONSOLE OUTPUT ---
# Original:  leave days pt staff?
# Rewritten: How many paid leave days are part-time employees entitled to?

11.2 HyDE — Hypothetical Document Embeddings

Instead of embedding the raw question, HyDE first generates a hypothetical answer, then embeds that. A hypothetical answer is far more similar to a real document chunk than a short question is.

# HyDE: Generate a hypothetical answer, embed it, use IT for retrieval

hyde_prompt = """
Write a short, factual paragraph that would answer this question 
if it appeared in a corporate HR handbook:

Question: How many sick days do part-time employees receive?
"""

hypothetical_answer = llm.predict(hyde_prompt)
print(f"Hypothetical: {hypothetical_answer}")

# Now embed the hypothetical answer instead of the original question
hyde_vector = embedder.embed_query(hypothetical_answer)
# Use hyde_vector for similarity search instead of query_vector

# --- CONSOLE OUTPUT ---
# Hypothetical: Part-time employees at the company are entitled to 
# 5 sick days per calendar year, prorated based on their contracted hours.

11.3 Re-ranking

Vector similarity retrieves broadly relevant chunks. A cross-encoder re-ranker then re-scores each chunk pair (query, chunk) with higher precision before passing them to the LLM.

from sentence_transformers import CrossEncoder

# Load a cross-encoder re-ranker (runs locally, free)
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

query = "How many leave days for part-time employees?"
retrieved_chunks = [
    "Part-time employees receive 7 days of annual leave.",
    "All employees must submit leave requests 2 weeks in advance.",
    "Full-time employees receive 15 days of annual leave per year.",
    "The company was founded in 2001 in Chennai."
]

# Score each (query, chunk) pair
pairs = [[query, chunk] for chunk in retrieved_chunks]
scores = reranker.predict(pairs)

# Sort by re-ranker score (descending)
ranked = sorted(zip(scores, retrieved_chunks), reverse=True)

print("Re-ranked results:")
for score, chunk in ranked:
    print(f"  Score: {score:.4f} | {chunk[:80]}")

# --- CONSOLE OUTPUT ---
# Re-ranked results:
#   Score: 9.8231 | Part-time employees receive 7 days of annual leave.
#   Score: 7.1044 | Full-time employees receive 15 days of annual leave per year.
#   Score: 4.2199 | All employees must submit leave requests 2 weeks in advance.
#   Score: 0.3122 | The company was founded in 2001 in Chennai.

11.4 Multi-Query Retrieval

Complex questions have multiple aspects. Multi-query retrieval generates several reformulations of the question and runs each as a separate retrieval, then deduplicates and merges the results.

from langchain.retrievers.multi_query import MultiQueryRetriever

# This automatically generates 3 query variations and merges results
multi_retriever = MultiQueryRetriever.from_llm(
    retriever=vector_db.as_retriever(search_kwargs={"k": 3}),
    llm=llm
)

unique_docs = multi_retriever.get_relevant_documents(
    "Tell me about leave policies for all employee types."
)
print(f"Documents retrieved via multi-query: {len(unique_docs)}")

# --- CONSOLE OUTPUT ---
# Documents retrieved via multi-query: 9

import os
from langchain.document_loaders import PyPDFLoader

answer = chain.run("How many annual leave days do part-time employees get?")

13. Evaluating RAG Systems

Deploying a RAG system without evaluation is dangerous. You need to measure both retrieval quality (did we fetch the right chunks?) and generation quality (did the LLM answer correctly from those chunks?).

Key RAG Metrics

MetricMeasuresHow to Interpret
FaithfulnessDoes the answer only contain claims supported by the retrieved context?1.0 = fully grounded, 0.0 = fully hallucinated
Answer RelevanceDoes the answer actually address the user's question?High score = answer is on-topic and complete
Context PrecisionOf the chunks retrieved, what fraction were actually useful?Low = retriever returned lots of noise/irrelevant chunks
Context RecallOf all relevant chunks that exist, what fraction were retrieved?Low = retriever missed important information
Answer CorrectnessSemantic similarity between generated answer and ground-truth answerRequires a labelled evaluation dataset

Code Example: Evaluation with Ragas

from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision
from datasets import Dataset

# Your test dataset: questions, ground truths, LLM answers, retrieved contexts
eval_data = {
    "question": [
        "How many leave days do part-time employees get?"
    ],
    "answer": [
        "Part-time employees receive 7 days of paid annual leave."
    ],
    "contexts": [
        ["Part-time employees receive 7 days of paid annual leave.",
         "All employees must file requests 2 weeks ahead."]
    ],
    "ground_truth": [
        "Part-time employees are entitled to 7 paid annual leave days per year."
    ]
}

dataset = Dataset.from_dict(eval_data)

results = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision]
)

print(results)

# --- CONSOLE OUTPUT ---
# {'faithfulness': 1.0000, 'answer_relevancy': 0.9743, 'context_precision': 0.8500}
💬 Discussion: How do you build a RAG evaluation dataset?

Creating a good evaluation (test) dataset is the hardest part of RAG evaluation. Best approach: (1) Take 50–100 representative questions users actually ask. (2) Have a domain expert write the correct "ground truth" answer for each. (3) Run your RAG system and compare outputs. (4) Track metrics over time as you change chunking strategies, embedding models, or prompts. Without this, you are flying blind.

15. Challenges and Best Practices

Building a RAG prototype is straightforward. Building a reliable, production-grade RAG system that handles real-world queries accurately is significantly harder.

Common Failure Modes

ProblemRoot CauseFix
Poor ChunkingCutting sentences mid-way destroys semantic meaning of the chunkUse Recursive splitter; always include overlap; test chunk quality manually
Irrelevant RetrievalWrong chunks returned due to surface-level similarity, not semantic matchBetter embedding model; metadata filtering; re-ranking; query rewriting
Lost in the MiddleLLM ignores chunks placed in the middle of a long contextLimit to top-3 chunks; put most relevant chunk first; use refine chain type
Stale Knowledge BaseDocuments updated but vector DB not re-indexedImplement automated sync pipeline; track document hashes to detect changes
Missing InformationAnswer exists in document but retriever didn't find itIncrease K; use multi-query retrieval; add hybrid search (keyword + vector)
Prompt InjectionMalicious content in documents overrides system instructionsSanitise document content; use strict system prompts; validate outputs

Best Practices Checklist

  • Use meaningful metadata: Tag every chunk with source file, section, date, department, and document version. This enables precise filtering and better citation.
  • Tune chunk size to your data: Short FAQs → smaller chunks (128–256 tokens). Dense technical documents → larger chunks (512–1024 tokens). Always test and measure.
  • Use hybrid search: Combine BM25 keyword search with vector similarity for robustness. Keyword search excels at exact matches (product codes, names); vector search handles meaning.
  • Enforce strict prompt boundaries: Instruct the LLM to answer ONLY from context and to say "I don't know" rather than guess. Never use high temperature for factual RAG.
  • Add a re-ranker: A lightweight cross-encoder re-ranker (e.g. ms-marco-MiniLM) applied to top-20 chunks before passing top-3 to the LLM significantly improves precision.
  • Build an eval dataset early: Collect 50+ real user questions with correct answers before going to production. Run evals after every pipeline change.
  • Instrument everything: Log every query, retrieved chunks, similarity scores, and generated answers. Analyse failed queries weekly to drive improvements.
  • Version your prompts: Treat prompts like code — version control, A/B test, and roll back when a new version degrades quality.

ingest.py — Build the Knowledge Base

"""
ingest.py
Run this script once to load documents and build the vector store.
Re-run whenever you add or update source documents.
"""

import os
import glob
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain.vectorstores import Chroma

os.environ["OPENAI_API_KEY"] = "your_openai_api_key_here"

DOCUMENTS_DIR  = "./documents"
CHROMA_PATH    = "./chroma_db"
CHUNK_SIZE     = 500
CHUNK_OVERLAP  = 50

def ingest_documents():
    all_chunks = []

    # Load all PDFs from the documents folder
    pdf_files = glob.glob(f"{DOCUMENTS_DIR}/*.pdf")
    print(f"Found {len(pdf_files)} PDF(s) to ingest.")

    for pdf_path in pdf_files:
        print(f"  Loading: {pdf_path}")
        loader = PyPDFLoader(pdf_path)
        pages  = loader.load()

        # Attach source filename as metadata
        for page in pages:
            page.metadata["source"] = os.path.basename(pdf_path)

        # Split into overlapping chunks
        splitter = RecursiveCharacterTextSplitter(
            chunk_size=CHUNK_SIZE,
            chunk_overlap=CHUNK_OVERLAP
        )
        chunks = splitter.split_documents(pages)
        all_chunks.extend(chunks)
        print(f"    → {len(chunks)} chunks created from {len(pages)} pages.")

    print(f"\nTotal chunks across all documents: {len(all_chunks)}")

    # Generate embeddings and persist to ChromaDB
    print("Generating embeddings and storing in ChromaDB...")
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vector_db  = Chroma.from_documents(
        documents=all_chunks,
        embedding=embeddings,
        persist_directory=CHROMA_PATH
    )
    vector_db.persist()
    print(f"Done. Vector store saved to: {CHROMA_PATH}")

if __name__ == "__main__":
    ingest_documents()

# --- CONSOLE OUTPUT ---
# Found 1 PDF(s) to ingest.
#   Loading: ./documents/handbook.pdf
#     → 84 chunks created from 42 pages.
# Total chunks across all documents: 84
# Generating embeddings and storing in ChromaDB...
# Done. Vector store saved to: ./chroma_db

app.py — Interactive Q&A System

"""
app.py
Interactive RAG Q&A system.
Run after ingest.py has built the vector store.
"""

import os
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.vectorstores import Chroma
from langchain.prompts import PromptTemplate
from langchain.chains import RetrievalQA

os.environ["OPENAI_API_KEY"] = "your_openai_api_key_here"

CHROMA_PATH = "./chroma_db"

# ── Load vector store ──────────────────────────────────────────
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vector_db  = Chroma(
    persist_directory=CHROMA_PATH,
    embedding_function=embeddings
)

# ── Custom RAG prompt ──────────────────────────────────────────
RAG_PROMPT = PromptTemplate(
    input_variables=["context", "question"],
    template="""You are a precise document assistant.
Answer the question using ONLY the context below.
If the answer is not in the context, say "I don't have that information."
Always mention the source document at the end of your answer.

CONTEXT:
{context}

QUESTION: {question}

ANSWER:"""
)

# ── Build the RAG chain ────────────────────────────────────────
llm = ChatOpenAI(model_name="gpt-4o", temperature=0)

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=vector_db.as_retriever(search_kwargs={"k": 4}),
    chain_type_kwargs={"prompt": RAG_PROMPT},
    return_source_documents=True
)

# ── Interactive Q&A loop ───────────────────────────────────────
print("=" * 60)
print("  RAG Document Q&A System — Type 'quit' to exit")
print("=" * 60)

while True:
    user_query = input("\nYour Question: ").strip()

    if user_query.lower() in ["quit", "exit", "q"]:
        print("Exiting. Goodbye!")
        break

    if not user_query:
        continue

    result = qa_chain({"query": user_query})

    print(f"\nAnswer: {result['result']}")
    print("\nSources:")
    seen = set()
    for doc in result["source_documents"]:
        src = doc.metadata.get("source", "Unknown")
        pg  = doc.metadata.get("page", "?")
        key = f"{src} p.{pg}"
        if key not in seen:
            print(f"  - {key}")
            seen.add(key)

# --- CONSOLE OUTPUT ---
# ============================================================
#   RAG Document Q&A System — Type 'quit' to exit
# ============================================================
#
# Your Question: How many sick days do part-time employees get?
#
# Answer: Part-time employees are entitled to 5 paid sick days per calendar year,
# as per the company's standard leave policy. This is prorated for employees 
# who join mid-year. [Source: handbook.pdf]
#
# Sources:
#   - handbook.pdf p.12
#   - handbook.pdf p.13
