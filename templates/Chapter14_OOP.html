2. Introduction to OOP

Object-Oriented Programming (OOP) is a programming paradigm based on the concept of objects. An object combines data (called attributes or properties) with behavior (called methods or functions). By modeling software after real-world entities, OOP enables developers to build complex, scalable, and maintainable systems.

Think of a Car as an analogy. A car has attributes like color, brand, and speed, and it has behaviors like start, accelerate, and brake. In OOP, you capture this by creating a Car class with the corresponding attributes and methods. Every individual car you create (a red Toyota, a blue Honda) is an instance (object) of that class.

The Four Pillars of OOP

  • Encapsulation — Bundling data and methods together, hiding internal details
  • Inheritance — A new class inherits attributes and methods from an existing class
  • Polymorphism — The same interface can behave differently depending on the object
  • Abstraction — Hiding complex implementation and exposing only what's necessary
Note: Most enterprise applications, ERP systems, banking software, hospital management systems, games, AI applications, and web platforms use OOP. Python fully supports OOP and treats everything — including integers, strings, and functions — as objects.

3. Procedural Programming vs OOP

Procedural programming executes a sequence of instructions one after another using functions. OOP organizes code around data and the objects that hold it. Here is a direct comparison:

Feature Procedural Programming Object-Oriented Programming
Focus Step-by-step instructions (functions) Data and objects (classes)
Data Security Data is global and exposed Data is hidden (Encapsulation)
Reusability Code reuse is difficult High reuse via Inheritance and Composition
Scalability Hard to maintain as logic grows Modular and easy to scale
Abstraction Limited abstraction capability Strong abstraction using classes and interfaces
Real-world modeling Poor mapping to real-world entities Natural mapping to real-world objects
Examples C, Pascal, early BASIC Python, Java, C++, C#

Side-by-Side Code Comparison

The same student record, written both ways:

# --- Procedural Style ---
name = "Alice"
roll_no = 101

def display_student(name, roll_no):
    print(f"Student: {name}, Roll No: {roll_no}")

display_student(name, roll_no)

# --- OOP Style ---
class Student:
    def __init__(self, name, roll_no):
        self.name = name
        self.roll_no = roll_no

    def display(self):
        print(f"Student: {self.name}, Roll No: {self.roll_no}")

s1 = Student("Alice", 101)
s1.display()

As the system grows (more fields, more logic), the OOP version scales far more cleanly because all student-related data and behavior stays inside the Student class.

4. Classes and Objects

A class is a user-defined blueprint or prototype from which objects are created. It defines a set of attributes and methods that characterize any object of that class. An object is a specific instance of a class — it occupies memory and holds its own state.

Defining a Class

Use the class keyword followed by the class name (by convention, use PascalCase):

class Student:
    def __init__(self, name, roll_no, grade):
        self.name = name          # Instance Variable
        self.roll_no = roll_no    # Instance Variable
        self.grade = grade        # Instance Variable

    def display(self):
        print(f"Name: {self.name} | Roll No: {self.roll_no} | Grade: {self.grade}")

Creating Objects (Instantiation)

# Creating instances (objects) of the Student class
s1 = Student("Alice", 101, "A")
s2 = Student("Bob",   102, "B")

s1.display()   # Name: Alice | Roll No: 101 | Grade: A
s2.display()   # Name: Bob   | Roll No: 102 | Grade: B

Accessing and Modifying Attributes

print(s1.name)       # Alice
s1.grade = "A+"      # Modify attribute directly
print(s1.grade)      # A+

The self Parameter

self refers to the current object instance. It is the first parameter of every instance method and is passed automatically by Python — you never pass it yourself when calling a method.

class Dog:
    def __init__(self, name):
        self.name = name      # 'self.name' belongs to this specific Dog

    def bark(self):
        print(f"{self.name} says: Woof!")

d1 = Dog("Rex")
d2 = Dog("Buddy")
d1.bark()   # Rex says: Woof!
d2.bark()   # Buddy says: Woof!
Note: self is a convention, not a keyword. You could technically name it anything, but always use self to keep your code readable and conventional.

Checking an Object's Type

s1 = Student("Alice", 101, "A")
print(type(s1))            # <class '__main__.Student'>
print(isinstance(s1, Student))  # True

5. Constructors and Variables

Constructors (__init__)

The __init__ method is a special method automatically called when an object is created. It initializes the object's attributes. A class can have only one __init__, but you can provide default values to make some arguments optional.

class Employee:
    def __init__(self, name, salary, department="General"):
        self.name = name
        self.salary = salary
        self.department = department

e1 = Employee("Alice", 50000, "Engineering")
e2 = Employee("Bob", 40000)           # Uses default department

print(e1.department)   # Engineering
print(e2.department)   # General

Instance Variables vs Class Variables

  • Instance Variables: Declared inside methods using self. They are unique to every object — each instance gets its own copy.
  • Class Variables: Declared inside the class but outside any method. They are shared across all instances of the class.
class Student:
    school = "ABC School"        # Class Variable — shared by all students
    student_count = 0            # Class Variable — tracks total students

    def __init__(self, name, roll_no):
        self.name = name         # Instance Variable — unique per student
        self.roll_no = roll_no   # Instance Variable — unique per student
        Student.student_count += 1

s1 = Student("Alice", 101)
s2 = Student("Bob", 102)

print(Student.school)        # ABC School
print(s1.school)             # ABC School  (accessible via instance too)
print(Student.student_count) # 2
Warning: If you assign to self.school on an instance, Python creates a new instance variable that shadows the class variable — it does not modify the shared class variable. Always modify class variables via the class name: Student.school = "XYZ School".
Type Declared Accessed via Shared?
Instance Variable Inside __init__ or other methods using self self.var or object.var No — unique per object
Class Variable Inside class body, outside methods ClassName.var or self.var Yes — all instances share it

Deleting Attributes and Objects

s1 = Student("Alice", 101)
del s1.roll_no        # Remove a specific attribute
del s1                # Remove the entire object

6. Methods

Methods define the behavior of objects. Python supports three types of methods inside a class:

Instance Methods

The most common type. They can access and modify instance attributes using self, and can also access class variables.

class Circle:
    PI = 3.14159          # Class variable

    def __init__(self, radius):
        self.radius = radius   # Instance variable

    def area(self):
        return Circle.PI * self.radius ** 2

    def perimeter(self):
        return 2 * Circle.PI * self.radius

c = Circle(5)
print(f"Area: {c.area():.2f}")        # Area: 78.54
print(f"Perimeter: {c.perimeter():.2f}")  # Perimeter: 31.42

Class Methods

Decorated with @classmethod. They receive the class itself (cls) as the first argument instead of an instance. Used to create alternative constructors or work with class-level state.

class Date:
    def __init__(self, day, month, year):
        self.day = day
        self.month = month
        self.year = year

    @classmethod
    def from_string(cls, date_string):
        """Alternative constructor: parse 'DD-MM-YYYY' string."""
        day, month, year = map(int, date_string.split("-"))
        return cls(day, month, year)     # Creates a new Date object

    def display(self):
        print(f"{self.day:02d}-{self.month:02d}-{self.year}")

d1 = Date(15, 8, 2024)
d2 = Date.from_string("01-01-2025")    # Using class method constructor
d1.display()   # 15-08-2024
d2.display()   # 01-01-2025

Static Methods

Decorated with @staticmethod. They have no access to self or cls. They behave like regular functions but live inside the class namespace because they are logically related to the class.

class MathUtils:
    @staticmethod
    def add(a, b):
        return a + b

    @staticmethod
    def is_even(n):
        return n % 2 == 0

    @staticmethod
    def factorial(n):
        if n == 0:
            return 1
        return n * MathUtils.factorial(n - 1)

print(MathUtils.add(3, 4))        # 7
print(MathUtils.is_even(10))      # True
print(MathUtils.factorial(5))     # 120
Method Type First Parameter Decorator Can Access
Instance Method self None Instance + class attributes
Class Method cls @classmethod Class attributes only
Static Method None @staticmethod Neither — acts like a utility function

7. Encapsulation

Encapsulation is the practice of bundling data (attributes) and the methods that operate on that data into a single unit (a class), and restricting direct access to some of those attributes. It protects an object's internal state from accidental or unauthorized modification.

Access Modifiers in Python

Python does not enforce access control at the language level, but uses naming conventions:

  • Public (name) — accessible from anywhere
  • Protected (_name) — conventionally internal; accessible but not meant to be used outside the class or subclasses
  • Private (__name) — name-mangled by Python, making it harder (but not impossible) to access from outside
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner          # Public
        self._account_no = "ACC001" # Protected (convention)
        self.__balance = balance    # Private (name-mangled)

    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount
            print(f"Deposited ₹{amount}. New balance: ₹{self.__balance}")
        else:
            print("Deposit amount must be positive.")

    def withdraw(self, amount):
        if 0 < amount <= self.__balance:
            self.__balance -= amount
            print(f"Withdrawn ₹{amount}. Remaining balance: ₹{self.__balance}")
        else:
            print("Invalid withdrawal amount or insufficient funds.")

    def get_balance(self):
        """Getter method — controlled read access."""
        return self.__balance


acc = BankAccount("Alice", 10000)
acc.deposit(5000)      # Deposited ₹5000. New balance: ₹15000
acc.withdraw(2000)     # Withdrawn ₹2000. Remaining balance: ₹13000
print(acc.get_balance())   # 13000

# Direct access to __balance raises AttributeError:
# print(acc.__balance)   # AttributeError!

Getters and Setters

Python's @property decorator lets you define getter and setter logic while keeping attribute-style syntax clean:

class Temperature:
    def __init__(self, celsius):
        self.__celsius = celsius

    @property
    def celsius(self):
        """Getter"""
        return self.__celsius

    @celsius.setter
    def celsius(self, value):
        """Setter with validation"""
        if value < -273.15:
            raise ValueError("Temperature cannot go below absolute zero!")
        self.__celsius = value

    @property
    def fahrenheit(self):
        """Computed property — no setter needed"""
        return (self.__celsius * 9/5) + 32

t = Temperature(25)
print(t.celsius)       # 25
print(t.fahrenheit)    # 77.0
t.celsius = 100
print(t.fahrenheit)    # 212.0

Benefits of Encapsulation

  • Data Security — prevents accidental changes to internal state
  • Controlled Access — validates data before accepting changes (e.g., balance can't go negative)
  • Improved Maintainability — internal implementation can change without affecting external code
  • Code Modularity — each class manages its own state independently

8. Inheritance

Inheritance allows a new class (called the child or derived class) to inherit attributes and methods from an existing class (called the parent or base class). This promotes code reuse and establishes an "is-a" relationship.

class Animal:
    def __init__(self, name, sound):
        self.name = name
        self.sound = sound

    def speak(self):
        print(f"{self.name} says {self.sound}")

    def breathe(self):
        print(f"{self.name} breathes air.")

class Dog(Animal):
    def fetch(self):
        print(f"{self.name} fetches the ball!")

class Cat(Animal):
    def purr(self):
        print(f"{self.name} purrs...")

d = Dog("Rex", "Woof")
c = Cat("Whiskers", "Meow")

d.speak()    # Rex says Woof  (inherited from Animal)
d.fetch()    # Rex fetches the ball!  (Dog-specific)
c.speak()    # Whiskers says Meow
c.purr()     # Whiskers purrs...

The super() Function

super() gives access to the parent class's methods. It is commonly used in __init__ to call the parent constructor before adding child-specific initialization:

class Vehicle:
    def __init__(self, brand, speed):
        self.brand = brand
        self.speed = speed

    def info(self):
        print(f"Brand: {self.brand} | Speed: {self.speed} km/h")

class ElectricVehicle(Vehicle):
    def __init__(self, brand, speed, battery_capacity):
        super().__init__(brand, speed)          # Call parent __init__
        self.battery_capacity = battery_capacity  # Add EV-specific attribute

    def info(self):
        super().info()                           # Call parent info()
        print(f"Battery: {self.battery_capacity} kWh")

ev = ElectricVehicle("Tesla", 250, 100)
ev.info()
# Brand: Tesla | Speed: 250 km/h
# Battery: 100 kWh

Types of Inheritance

Type Description Example
Single One child inherits from one parent Dog(Animal)
Multiple One child inherits from multiple parents FlyingFish(Bird, Fish)
Multilevel A chain of inheritance: A → B → C Puppy(Dog), Dog(Animal)
Hierarchical Multiple children share one parent Dog(Animal), Cat(Animal)
Hybrid Combination of two or more types Mix of multiple + multilevel

Multiple Inheritance Example

class Flyable:
    def fly(self):
        print("I can fly!")

class Swimmable:
    def swim(self):
        print("I can swim!")

class Duck(Flyable, Swimmable):
    def quack(self):
        print("Quack!")

donald = Duck()
donald.fly()    # I can fly!
donald.swim()   # I can swim!
donald.quack()  # Quack!
Warning: Multiple inheritance can lead to the "Diamond Problem" — ambiguity when two parent classes share a common ancestor. Python resolves this using the Method Resolution Order (MRO). Use ClassName.__mro__ or help(ClassName) to inspect it.

Checking Inheritance Relationships

print(issubclass(Dog, Animal))    # True
print(issubclass(Animal, Dog))    # False
print(isinstance(d, Animal))      # True  (Dog IS-A Animal)

9. Polymorphism

Polymorphism means "many forms." In OOP, it means that the same method name can behave differently depending on the object it is called on. This makes code more flexible and extensible.

Method Overriding (Runtime Polymorphism)

A child class provides its own implementation of a method already defined in the parent class:

class Shape:
    def area(self):
        return 0

class Rectangle(Shape):
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height

class Circle(Shape):
    PI = 3.14159

    def __init__(self, radius):
        self.radius = radius

    def area(self):
        return Circle.PI * self.radius ** 2

class Triangle(Shape):
    def __init__(self, base, height):
        self.base = base
        self.height = height

    def area(self):
        return 0.5 * self.base * self.height

shapes = [Rectangle(4, 5), Circle(3), Triangle(6, 8)]

for shape in shapes:
    print(f"{shape.__class__.__name__}: area = {shape.area():.2f}")
# Rectangle: area = 20.00
# Circle: area = 28.27
# Triangle: area = 24.00

Duck Typing

Python uses duck typing: if an object has the right method, Python doesn't care what class it is. "If it walks like a duck and quacks like a duck, it is a duck."

class Dog:
    def sound(self):
        print("Bark!")

class Cat:
    def sound(self):
        print("Meow!")

class Robot:
    def sound(self):
        print("Beep boop!")

def make_sound(animal):
    animal.sound()    # Works for any object with a sound() method

for obj in [Dog(), Cat(), Robot()]:
    make_sound(obj)
# Bark!
# Meow!
# Beep boop!

Operator Overloading (compile-time polymorphism)

Python allows you to redefine the behaviour of operators (+, -, ==, etc.) for your custom classes using special methods:

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __str__(self):
        return f"Vector({self.x}, {self.y})"

v1 = Vector(1, 2)
v2 = Vector(3, 4)
v3 = v1 + v2      # Calls __add__
print(v3)         # Vector(4, 6)

10. Abstraction

Abstraction means hiding complex internal details and only exposing what is necessary to the outside world. It forces a clean contract: any class that inherits from an abstract class must implement the abstract methods — otherwise Python raises a TypeError.

In Python, abstraction is achieved using the abc module (Abstract Base Classes):

from abc import ABC, abstractmethod

class Shape(ABC):
    """Abstract base class — cannot be instantiated directly."""

    @abstractmethod
    def area(self):
        pass

    @abstractmethod
    def perimeter(self):
        pass

    def describe(self):
        """Concrete method — shared by all shapes."""
        print(f"I am a {self.__class__.__name__} with area {self.area():.2f}")


class Rectangle(Shape):
    def __init__(self, w, h):
        self.w = w
        self.h = h

    def area(self):
        return self.w * self.h

    def perimeter(self):
        return 2 * (self.w + self.h)


class Circle(Shape):
    def __init__(self, r):
        self.r = r

    def area(self):
        return 3.14159 * self.r ** 2

    def perimeter(self):
        return 2 * 3.14159 * self.r


r = Rectangle(4, 6)
c = Circle(5)
r.describe()   # I am a Rectangle with area 24.00
c.describe()   # I am a Circle with area 78.54

# shape = Shape()  # TypeError: Can't instantiate abstract class Shape
Note: Abstract classes can contain both abstract methods (which must be overridden) and concrete methods (which can be used as-is or overridden optionally). This lets you define shared behavior alongside enforced contracts.

Abstract Class vs Interface (Python context)

Feature Abstract Class (Python ABC) Interface-like class (all abstract)
Can have concrete methods Yes No (by design)
Can have instance variables Yes No (by design)
Enforces method implementation Yes (for @abstractmethod) Yes
Python keyword ABC, @abstractmethod Same (all methods marked abstract)

11. Magic Methods (Dunder Methods)

Magic methods (also called dunder methods — double underscore) are special methods that Python calls automatically in certain situations. They allow your custom objects to integrate with Python's built-in operations and syntax.

Common Magic Methods

Method Triggered By Purpose
__init__ obj = MyClass() Initialize the object
__str__ print(obj) / str(obj) Human-readable string representation
__repr__ repr(obj) / REPL display Developer-facing debug string
__len__ len(obj) Return the "length" of the object
__add__ obj1 + obj2 Addition operator
__eq__ obj1 == obj2 Equality comparison
__lt__ obj1 < obj2 Less-than comparison
__del__ Object is garbage collected Cleanup / destructor
__getitem__ obj[key] Index / key access
__contains__ x in obj Membership test

Full Example — A Custom Book Class

class Book:
    def __init__(self, title, author, pages):
        self.title = title
        self.author = author
        self.pages = pages

    def __str__(self):
        return f'"{self.title}" by {self.author}'

    def __repr__(self):
        return f"Book(title={self.title!r}, author={self.author!r}, pages={self.pages})"

    def __len__(self):
        return self.pages

    def __eq__(self, other):
        return self.title == other.title and self.author == other.author

    def __lt__(self, other):
        return self.pages < other.pages

b1 = Book("Python Basics", "John", 300)
b2 = Book("Advanced Python", "Jane", 450)
b3 = Book("Python Basics", "John", 300)

print(b1)              # "Python Basics" by John
print(repr(b1))        # Book(title='Python Basics', author='John', pages=300)
print(len(b1))         # 300
print(b1 == b3)        # True  (same title + author)
print(b1 < b2)         # True  (300 < 450)

books = [b2, b1]
books.sort()           # Sorts using __lt__
print([str(b) for b in books])
# ['"Python Basics" by John', '"Advanced Python" by Jane']

Context Manager Magic: __enter__ and __exit__

Implementing these two methods lets your class work with Python's with statement — perfect for managing resources like files or database connections:

class FileManager:
    def __init__(self, filename, mode):
        self.filename = filename
        self.mode = mode
        self.file = None

    def __enter__(self):
        self.file = open(self.filename, self.mode)
        return self.file

    def __exit__(self, exc_type, exc_val, exc_tb):
        if self.file:
            self.file.close()
        print("File closed.")
        return False   # Don't suppress exceptions

with FileManager("data.txt", "w") as f:
    f.write("Hello, OOP!")
# File closed.   (automatic cleanup)

12. Composition and Aggregation

Inheritance models an "is-a" relationship. Composition and Aggregation model a "has-a" relationship — they describe how objects are built from other objects.

Composition — Strong Ownership

The contained object's lifecycle is managed by the container. If the container is destroyed, so is the contained object. Example: a Car owns its Engine.

class Engine:
    def __init__(self, horsepower):
        self.horsepower = horsepower

    def start(self):
        print(f"Engine with {self.horsepower}HP started.")

class Car:
    def __init__(self, brand, horsepower):
        self.brand = brand
        self.engine = Engine(horsepower)   # Car OWNS the Engine

    def start(self):
        print(f"{self.brand} car:")
        self.engine.start()

my_car = Car("Toyota", 120)
my_car.start()
# Toyota car:
# Engine with 120HP started.

Aggregation — Weak Ownership

The contained object exists independently. It is passed in from outside and can outlive the container. Example: a Department has Employees, but employees can exist without the department.

class Employee:
    def __init__(self, name):
        self.name = name

class Department:
    def __init__(self, dept_name):
        self.dept_name = dept_name
        self.employees = []          # Aggregated — not created here

    def add_employee(self, employee):
        self.employees.append(employee)

    def list_employees(self):
        print(f"Department: {self.dept_name}")
        for emp in self.employees:
            print(f"  - {emp.name}")

e1 = Employee("Alice")
e2 = Employee("Bob")

dept = Department("Engineering")
dept.add_employee(e1)
dept.add_employee(e2)
dept.list_employees()
# Department: Engineering
#   - Alice
#   - Bob

# e1 and e2 still exist even if dept is deleted
del dept
print(e1.name)   # Alice — still accessible
Feature Composition Aggregation
Relationship strength Strong (owns) Weak (uses/has)
Lifecycle Part dies with the container Part can exist independently
Object creation Created inside the class Passed in from outside
UML notation Filled diamond ◆ Open diamond ◇
Example Car ◆ Engine Department ◇ Employee
Design Tip: Prefer composition over inheritance when possible. Inheritance creates tight coupling between classes; composition is more flexible and easier to test.

13. SOLID Principles

SOLID is a set of five design principles that make software more understandable, flexible, and maintainable. They were popularized by Robert C. Martin ("Uncle Bob").

S — Single Responsibility Principle (SRP)

A class should have one and only one reason to change. Each class should do one job.

# Bad: One class doing too much
class Student:
    def enroll(self): ...
    def calculate_grade(self): ...
    def send_email(self): ...        # Unrelated responsibility!
    def save_to_database(self): ...  # Unrelated responsibility!

# Good: Responsibilities separated
class Student:
    def enroll(self): ...
    def calculate_grade(self): ...

class EmailService:
    def send_email(self, student): ...

class StudentRepository:
    def save(self, student): ...

O — Open/Closed Principle (OCP)

Classes should be open for extension but closed for modification. Add new behavior by creating new classes, not by changing existing ones.

from abc import ABC, abstractmethod

class Discount(ABC):
    @abstractmethod
    def apply(self, price):
        pass

class NoDiscount(Discount):
    def apply(self, price):
        return price

class TenPercentDiscount(Discount):
    def apply(self, price):
        return price * 0.90

class SeasonalDiscount(Discount):     # Extended — no existing code modified
    def apply(self, price):
        return price * 0.75

def checkout(price, discount: Discount):
    return discount.apply(price)

print(checkout(1000, TenPercentDiscount()))  # 900.0
print(checkout(1000, SeasonalDiscount()))    # 750.0

L — Liskov Substitution Principle (LSP)

Objects of a subclass should be replaceable with objects of the parent class without breaking the program. A child should extend the parent, not restrict or contradict it.

class Bird:
    def move(self):
        print("Moving...")

class Sparrow(Bird):
    def move(self):
        print("Flying!")       # Extends behavior — OK

class Penguin(Bird):
    def move(self):
        print("Swimming!")     # Still moves — LSP respected

def let_bird_move(bird: Bird):
    bird.move()

let_bird_move(Sparrow())   # Flying!
let_bird_move(Penguin())   # Swimming!

I — Interface Segregation Principle (ISP)

Clients should not be forced to depend on methods they do not use. Create small, focused interfaces rather than one large general-purpose one.

from abc import ABC, abstractmethod

# Bad: One big interface forces all classes to implement everything
# class Worker(ABC):
#     def work(self): ...
#     def eat(self): ...     # Robots don't eat!

# Good: Separate interfaces
class Workable(ABC):
    @abstractmethod
    def work(self): pass

class Eatable(ABC):
    @abstractmethod
    def eat(self): pass

class HumanWorker(Workable, Eatable):
    def work(self): print("Human working...")
    def eat(self): print("Human eating...")

class RobotWorker(Workable):
    def work(self): print("Robot working...")
    # No eat() — robots don't need it!

D — Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces/abstract classes).

from abc import ABC, abstractmethod

class Database(ABC):
    @abstractmethod
    def save(self, data): pass

class MySQLDatabase(Database):
    def save(self, data):
        print(f"Saving to MySQL: {data}")

class MongoDatabase(Database):
    def save(self, data):
        print(f"Saving to MongoDB: {data}")

class UserService:
    def __init__(self, db: Database):   # Depends on abstraction, not concrete class
        self.db = db

    def create_user(self, name):
        self.db.save(name)

# Swap databases without changing UserService
service = UserService(MySQLDatabase())
service.create_user("Alice")   # Saving to MySQL: Alice

service = UserService(MongoDatabase())
service.create_user("Bob")     # Saving to MongoDB: Bob
Letter Principle One-line Rule
SSingle ResponsibilityOne class, one job
OOpen/ClosedExtend without modifying
LLiskov SubstitutionSubclasses must honour parent's contract
IInterface SegregationSmall, focused interfaces
DDependency InversionDepend on abstractions, not concretions

Project 1: Bank Account System

A fully working bank account system with deposits, withdrawals, transaction history, and account types:

class BankAccount:
    def __init__(self, account_no, owner, balance=0):
        self.account_no = account_no
        self.owner = owner
        self.__balance = balance
        self.__transactions = []

    def deposit(self, amount):
        if amount <= 0:
            print("Amount must be positive.")
            return
        self.__balance += amount
        self.__transactions.append(f"Deposit: +₹{amount}")
        print(f"₹{amount} deposited. Balance: ₹{self.__balance}")

    def withdraw(self, amount):
        if amount <= 0:
            print("Amount must be positive.")
        elif amount > self.__balance:
            print("Insufficient funds.")
        else:
            self.__balance -= amount
            self.__transactions.append(f"Withdrawal: -₹{amount}")
            print(f"₹{amount} withdrawn. Balance: ₹{self.__balance}")

    def get_balance(self):
        return self.__balance

    def print_statement(self):
        print(f"\n--- Statement for {self.owner} ({self.account_no}) ---")
        for t in self.__transactions:
            print(" ", t)
        print(f"  Current Balance: ₹{self.__balance}\n")


class SavingsAccount(BankAccount):
    INTEREST_RATE = 0.04

    def add_interest(self):
        interest = self.get_balance() * SavingsAccount.INTEREST_RATE
        self.deposit(interest)
        print(f"Interest of ₹{interest:.2f} added.")


# --- Usage ---

acc.withdraw(2000)

acc.print_statement()

class Book:
    def __init__(self, isbn, title, author):
        self.isbn = isbn
        self.title = title
        self.author = author
        self.is_available = True

    def __str__(self):
        status = "Available" if self.is_available else "Borrowed"
        return f'[{self.isbn}] "{self.title}" by {self.author} — {status}'


class Member:
    def __init__(self, member_id, name):
        self.member_id = member_id
        self.name = name
        self.borrowed_books = []

    def borrow(self, book):
        if not book.is_available:
            print(f'"{book.title}" is not available.')
            return
        book.is_available = False
        self.borrowed_books.append(book)
        print(f'{self.name} borrowed "{book.title}".')

    def return_book(self, book):
        if book in self.borrowed_books:
            book.is_available = True
            self.borrowed_books.remove(book)
            print(f'{self.name} returned "{book.title}".')
        else:
            print("You didn't borrow this book.")


class Library:
    def __init__(self, name):
        self.name = name
        self.books = []

    def add_book(self, book):
        self.books.append(book)

    def search(self, keyword):
        results = [b for b in self.books if keyword.lower() in b.title.lower()]
        if results:
            for b in results:
                print(b)
        else:
            print("No books found.")

class Subject:
    def __init__(self, name, marks, max_marks=100):
        self.name = name
        self.marks = marks
        self.max_marks = max_marks

    def grade(self):
        pct = (self.marks / self.max_marks) * 100
        if pct >= 90: return "A+"
        elif pct >= 80: return "A"
        elif pct >= 70: return "B"
        elif pct >= 60: return "C"
        elif pct >= 50: return "D"
        else: return "F"


class Student:
    def __init__(self, roll_no, name):
        self.roll_no = roll_no
        self.name = name
        self.__subjects = []

    def add_subject(self, subject):
        self.__subjects.append(subject)

    def report_card(self):
        print(f"\n===== Report Card: {self.name} (Roll: {self.roll_no}) =====")
        total = 0
        for s in self.__subjects:
            print(f"  {s.name:20s} {s.marks:3}/{s.max_marks}  Grade: {s.grade()}")
            total += s.marks
        avg = total / len(self.__subjects) if self.__subjects else 0
        print(f"  {'Total Average':20s} {avg:.1f}%")
        print("=" * 50)

16. Common Errors in OOP

1. Missing self Parameter

# Wrong
class Dog:
    def bark():       # Missing self!
        print("Woof")

# Correct
class Dog:
    def bark(self):
        print("Woof")
Error: TypeError: bark() takes 0 positional arguments but 1 was given — Python passes the instance automatically, but your method didn't declare self to receive it.

2. Not Calling the Parent Constructor

# Wrong — parent attributes never initialized
class Animal:
    def __init__(self, name):
        self.name = name

class Dog(Animal):
    def __init__(self, name, breed):
        self.breed = breed    # Forgot super().__init__(name)!

d = Dog("Rex", "Lab")
print(d.name)   # AttributeError: 'Dog' object has no attribute 'name'

# Correct
class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)   # Initialize parent first
        self.breed = breed

3. Confusing Class Variables with Instance Variables

class Counter:
    count = 0       # Class variable

    def increment(self):
        self.count += 1   # Creates an INSTANCE variable! Class variable unchanged.

c1 = Counter()
c1.increment()
print(c1.count)       # 1 (instance variable)
print(Counter.count)  # 0 (class variable unchanged!)

# Correct way to modify class variable:
class Counter:
    count = 0

    def increment(self):
        Counter.count += 1   # Modify class variable correctly

4. Incorrect Inheritance Hierarchy

# Wrong — hierarchy reversed
class Dog(Puppy):    # Puppy is a type of Dog, not the other way!
    pass

# Correct
class Dog:
    pass

class Puppy(Dog):    # Puppy IS-A Dog
    pass

5. Modifying Private Attributes Directly

class Account:
    def __init__(self):
        self.__balance = 1000

acc = Account()
# Wrong — bypasses validation
acc.__balance = -5000   # This actually creates a NEW public attribute!
print(acc.__balance)    # -5000 (but the real private one is still 1000)

# Correct — use provided setter methods or @property
Warning: Python applies name mangling to private attributes: __balance becomes _Account__balance internally. Accessing acc.__balance from outside creates a new public attribute — it does not access the private one.

6. Forgetting @abstractmethod Leads to No Enforcement

from abc import ABC

class Shape(ABC):
    def area(self):   # Not marked @abstractmethod!
        pass

class Rectangle(Shape):
    pass              # No override — no error raised

# Python allows this silently. Always use @abstractmethod for enforcement.

Quick Reference: Error Checklist

Error Symptom Fix
Missing self TypeError: takes 0 positional arguments but 1 was given Add self as first parameter
No parent super().__init__() AttributeError on parent attributes Call super().__init__(...) in child
Wrong class variable update Class variable doesn't change; instance gets shadow copy Use ClassName.var to update class variable
Reversed inheritance Logic errors; wrong methods resolved Ensure child IS-A parent, not vice versa
Modifying private via obj.__attr Silently creates new attribute; real private unchanged Use getters/setters or @property
Abstract method not decorated No enforcement; subclass can skip implementation silently Always use @abstractmethod
